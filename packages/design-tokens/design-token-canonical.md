# repo/design-system/design-system-canonical.md

packages/design-tokens/ — Final Source of Truth

Purpose

packages/design-tokens is the single source of truth for visual primitives and shared tokenized design decisions across the monorepo.

It exists to answer one question cleanly:

What are the canonical visual primitives of the platform, and how are they exposed consistently across apps, packages, CSS, Tailwind v4, and Vanilla Extract?

This package is design-token infrastructure.
It is not a component library.
It is not app styling.
It is not a theme playground.

⸻

Role in the monorepo

This package sits low in the dependency graph and is shared by multiple surfaces.

It supports:
	•	Astro public/marketing surfaces
	•	Next.js product/app surfaces
	•	@repo/ui
	•	CSS consumers
	•	Tailwind v4 consumers
	•	Vanilla Extract consumers

It must remain:
	•	framework-agnostic
	•	stable
	•	explicit
	•	hard to bypass
	•	easy for agentic coding tools to extend safely

This package should become one of the lowest-churn packages in the repo once stabilized.

⸻

What this package owns

packages/design-tokens owns:
	•	raw token source files
	•	semantic token mapping
	•	theme overlay definitions
	•	generated CSS custom properties
	•	generated Tailwind v4 bridge variables
	•	generated TS token metadata
	•	Vanilla Extract token contract
	•	token validation rules
	•	token generation scripts
	•	package-level tests that prevent token drift

⸻

What this package does not own

packages/design-tokens does not own:
	•	UI components
	•	React/Astro component props
	•	app-specific styles
	•	page-specific styles
	•	feature-specific design hacks
	•	business/runtime theming logic
	•	CMS theming config
	•	per-client one-off token systems
	•	arbitrary utility classes
	•	Tailwind app configuration
	•	marketing-block-specific presentation rules

Those belong elsewhere, primarily:
	•	@repo/ui
	•	app-level CSS
	•	app-specific feature code

⸻

Hard decisions locked for this package

1. Single source of truth

All canonical design tokens originate here.

Apps and other packages must not invent parallel token systems.

⸻

2. Conservative DTCG subset only

This package uses DTCG-style authoring, but not the full latest DTCG surface.

Allowed authored keys
	•	$value
	•	$type
	•	$description
	•	$deprecated

Allowed reference syntax
	•	curly-brace aliases only
Example: {core.color.neutral.950}

Forbidden in v1 authored token files
	•	$root
	•	$extends
	•	JSON Pointer $ref
	•	arbitrary $extensions
	•	token/group shape mixing
	•	hybrid custom metadata structures that bypass the package contract

This is deliberate. The token files must be simple, predictable, and tool-safe.

⸻

3. Folder structure is for humans, not semantics

The filesystem layout exists for authoring clarity only.

Semantics are enforced by:
	•	the token object shape
	•	$type
	•	naming conventions
	•	custom validation
	•	custom transforms
	•	tests

The build must not assume that folder names alone define meaning.

⸻

4. Theme activation is standardized

The only canonical theme selectors are:
	•	:root
	•	html[data-theme="light"]
	•	html[data-theme="dark"]
	•	html[data-brand="firm"]
	•	html[data-brand="platform"]

No .dark class.
No app-specific selector patterns.
No alternate brand-selector conventions.

⸻

5. Tailwind v4 consumes tokens

Tailwind is a consumer of this package, not a co-owner of token truth.

tailwind.css is generated by this package and acts as a bridge from canonical package variables to Tailwind theme-variable namespaces.

Tailwind must not become the source of truth for:
	•	colors
	•	spacing
	•	radii
	•	typography
	•	breakpoints
	•	shadows

⸻

6. Vanilla Extract consumes the same global vars

Vanilla Extract is a typed consumer of the same CSS variable contract used by plain CSS and Tailwind.

This package uses a global contract approach, not a second VE-owned theming system.

Vanilla Extract does not become an alternate source of theme values.

⸻

7. Global token CSS is imported once per app

Each app imports package global CSS once at the root shell.
	•	Next imports from the root layout/global stylesheet path
	•	Astro imports from the shared page/layout shell

No feature-level CSS imports of package global token files.

⸻

8. New colors default to OKLCH

For new core palette authoring:
	•	prefer oklch()
	•	allow hex only when required for interoperability or exact brand handoff

This package should follow the project’s modern-browser baseline.

⸻

9. Build warnings fail

Token validation/generation is strict.

The build must fail on:
	•	unresolved references
	•	duplicate generated variable collisions
	•	illegal cross-layer references
	•	invalid theme overrides
	•	invalid file shapes
	•	drift in generated artifacts

⸻

10. Generated artifacts are committed

src/generated/* is:
	•	machine-owned
	•	committed
	•	reviewable in PRs
	•	never edited by hand

This is an anti-drift decision for agentic coding workflows.

⸻

11. No sprinkles in v1

sprinkles.ts is intentionally omitted in v1.

Reasons:
	•	it creates a second styling API too early
	•	the repo already has Tailwind v4 and VE
	•	this package should stay narrow and boring

⸻

12. Component tokens stay rare

Only these component token files exist in v1:
	•	button.json
	•	input.json
	•	card.json

Not included in v1:
	•	modal
	•	badge
	•	tabs
	•	nav
	•	hero
	•	marketing sections
	•	per-client sections
	•	arbitrary feature components

⸻

13. No tenant token files in v1

Tenant theming is documented, not implemented as token files.

There are no tenant theme JSON files in v1.

The package includes only tenant theming governance documentation for future use.

Design philosophy

1. Semantic-first consumption

Most consumers should use semantic tokens, not raw primitives.

Good:
	•	semantic.text.default
	•	semantic.surface.canvas
	•	semantic.spacing.stack.md
	•	semantic.typography.body.md

Weaker:
	•	core.color.blue.500
	•	core.spacing.4
	•	core.radius.lg

Core tokens are infrastructure.
Semantic tokens are the system API.

⸻

2. Tokens are layered

The package is built in layers:
	•	core: raw primitives
	•	semantic: meaning-based defaults for the default light system
	•	themes: overlays on semantic meaning
	•	component: sparse system-level component contracts

Without layering, token systems become unmaintainable.

⸻

3. Themes are overlays, not forks

Themes may override approved semantic surfaces.

Themes may not create incompatible parallel systems.

No theme may:
	•	redefine core primitives
	•	create its own spacing scale
	•	create its own typography scale
	•	fork component geometry
	•	introduce feature-level values

⸻

4. Naming is by intent, not implementation trivia

Names must survive:
	•	palette shifts
	•	rebrands
	•	theme expansion
	•	visual refinements

Good:
	•	semantic.text.default
	•	semantic.border.subtle
	•	semantic.typography.heading.lg
	•	semantic.surface.elevated

Bad:
	•	gray-700
	•	navy-accent
	•	border-light-2
	•	card-radius

⸻

5. Tokens are cross-framework infrastructure

This package must work cleanly for:
	•	Astro
	•	Next.js
	•	CSS
	•	Tailwind v4
	•	Vanilla Extract
	•	shared packages

So the package stays framework-agnostic.

⸻

Theme composition model

Composition order in generated CSS is:
	1.	default light semantic values
	2.	dark overlay
	3.	brand overlay

So:
	•	light = semantic defaults
	•	dark = semantic defaults + dark overrides
	•	dark + brand = semantic defaults + dark overrides + brand overrides

Brand overlays must work acceptably in both light and dark in v1.

No brand-specific dark fork in v1.

Naming contract

Token path naming

Canonical path patterns look like:
	•	core.color.neutral.0
	•	core.spacing.4
	•	semantic.text.default
	•	semantic.surface.canvas
	•	component.button.height.md

⸻

CSS variable naming

Canonical variable naming is derived from token paths using this locked prefix:
	•	--repo-

Examples:
	•	core.color.neutral.0 → --repo-core-color-neutral-0
	•	semantic.text.default → --repo-semantic-text-default
	•	component.button.height.md → --repo-component-button-height-md

Apps and packages must not invent alternate token variable prefixes.

⸻

Layer dependency rules

These rules are hard.

Allowed references
	•	core → core
	•	semantic → core, semantic
	•	component → core, semantic
	•	themes → semantic override targets only, using approved references

Forbidden references
	•	core → semantic/component/themes
	•	semantic → component/themes
	•	component → component/themes
	•	themes → core overrides
	•	any upward reference across layer boundaries

This keeps the graph understandable and safe.

⸻

Light/dark/brand model

Default light system

The default light system is defined directly by the semantic/* files.

There is no tokens/themes/base/light.json.

This is deliberate to avoid duplicated light-theme truth.

⸻

Dark system

tokens/themes/base/dark.json contains only differences from the default semantic light system.

⸻

Brand overlays

firm.json and platform.json contain only approved semantic overrides.

Brand overlays must remain narrow.

⸻

App consumption contract

Each app should have one root stylesheet.

That stylesheet should import, in this order:

@import "tailwindcss";
@import "@repo/design-tokens/global.css";
@import "@repo/design-tokens/tailwind.css";

Then app-global CSS may follow.

Rules
	•	apps import token CSS once
	•	components do not import package CSS directly
	•	features do not import generated files directly
	•	VE consumers still need global token CSS imported once

Build order

Generation flow
	1.	validate authored token files against schemas
	2.	run custom structural validation
	3.	build generated artifacts with Style Dictionary
	4.	write src/generated/*
	5.	compile TS entrypoints
	6.	copy generated CSS assets into dist
	7.	run tests

Definition of done

packages/design-tokens is done when:
	•	all canonical tokens originate here
	•	layer boundaries are enforced by validation and tests
	•	the default light semantic contract is complete
	•	dark and brand overlays generate correctly
	•	global.css and tailwind.css are stable app imports
	•	Tailwind consumes the package token bridge
	•	VE consumes the same global variable contract
	•	generated artifacts are deterministic and committed
	•	apps do not need local parallel token systems
	•	the README is strong enough for an agent to extend safely without drift
